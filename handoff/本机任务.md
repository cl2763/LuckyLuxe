## 待执行任务(数据迁移批)

### 任务 1:提交并推送上线安全批
1. `git status` 确认不含 `local-data/`、`*.sqlite`、`.env`、初始密码文件。
2. `git add -A && git commit -m "Production safety: gate demo admin login by env, secure DB import endpoint, migration prep tool"`
3. `git push origin main`(这会触发 Railway 自动部署,云端演示登录将被禁用——预期行为)。
4. 等待约 90 秒,`curl -s https://www.luckyluxeatelier.com/health` 确认服务恢复 200。

### 任务 2:开启云端导入开关
1. `railway variables --set "ALLOW_DB_IMPORT=true"`(会触发一次重新部署,等待服务恢复 /health 200)。

### 任务 3:本地准备迁移库
1. 确认 4128 无进程(前置条件)。
2. `node tools/prepare-migration.mjs apps/api/local-data/lucky-luxe.sqlite /tmp/lucky-luxe-migrate.sqlite`
3. 把脚本输出的**清理报告和保留数据快照数字**完整记入结果文件(迁移后要逐项核对)。
4. 预期输出体积约 3-5MB;若超过 50MB,停止并记入结果文件。

### 任务 4:上传到云端
1. 从 `railway variables --kv` 取 OWNER_DEMO_TOKEN 的值存入 shell 变量(**不要打印**)。
2. `curl -s -X POST -H "authorization: Bearer $TOKEN" -H "x-confirm-import: yes" --data-binary @/tmp/lucky-luxe-migrate.sqlite https://www.luckyluxeatelier.com/admin/ops/import-db`
3. 预期返回 `{"staged":true,...}`;服务会自动重启应用导入。等待约 60 秒。

### 任务 5:云端核对
1. `curl -s https://www.luckyluxeatelier.com/health` → 200。
2. 用同一 token 请求 `/admin/customers` 和 `/admin/bookings`,统计条数,与任务 3 快照中的 users/bookings 数字核对,一致与否记入结果文件。
3. 请求 `/admin/finance/lock-status`,预期 `{"configured":true}`(财务密码随库迁移)。

### 任务 6:关闭导入开关 + 清理
1. `railway variables --set "ALLOW_DB_IMPORT=false"`
2. `rm /tmp/lucky-luxe-migrate.sqlite`
3. 尝试用 CLI 删除自定义域名 app.luckyluxeatelier.com(若 CLI 不支持,记入结果文件留给网页操作)。
